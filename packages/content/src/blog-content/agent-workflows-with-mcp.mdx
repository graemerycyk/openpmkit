# How to design agent workflows with MCP connectors (and why it matters)

AI agents are only as useful as the tools they can access. An agent that can't read your Jira tickets, search your Slack history, or query your analytics is just a chatbot with ambition.

MCP (Model Context Protocol) solves this problem. It's an open standard for connecting AI agents to external tools and data sources, providing a consistent interface that works across different AI systems and enterprise applications.

This guide explains how to design agent workflows using MCP connectors, with practical patterns for enterprise PM tools.

## What is MCP?

MCP (Model Context Protocol) is a standardized way for AI agents to interact with external systems. Think of it as a universal adapter that lets any AI system connect to any tool.

### The Problem MCP Solves

Before MCP, connecting an AI agent to enterprise tools meant:

- **Custom integrations** for each tool (Jira API, Slack API, Gong API...)
- **Bespoke authentication** handling per system
- **Different data formats** requiring transformation
- **N×M complexity** as you add more agents and tools

With MCP:

- **Standardized interface** for all tools
- **Consistent authentication** patterns
- **Unified data formats** across systems
- **N+M complexity** (add tools or agents independently)

### MCP Architecture

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   AI Agent      │     │   MCP Server    │     │  External Tool  │
│  (Claude, etc.) │────▶│  (Connector)    │────▶│  (Jira, Slack)  │
└─────────────────┘     └─────────────────┘     └─────────────────┘
        │                       │                       │
        │   Tool Discovery      │   API Calls           │
        │   Tool Execution      │   Authentication      │
        │   Result Handling     │   Data Transform      │
        └───────────────────────┴───────────────────────┘
```

An MCP server exposes **tools** that agents can discover and call. Each tool has:

- **Name**: What the tool is called (e.g., `search_issues`)
- **Description**: What the tool does
- **Input schema**: What parameters it accepts
- **Output schema**: What it returns

## Designing Agent Workflows with MCP

### Workflow Pattern 1: Data Aggregation

**Use case**: Daily briefs, VoC reports, meeting prep

**Pattern**: Agent calls multiple read tools, synthesizes results

```typescript
// Workflow: Daily Brief Generation

// Step 1: Gather data from multiple sources
const slackMessages = await mcp.call('slack', 'get_messages', {
  channels: ['product-feedback', 'customer-success'],
  since: yesterday,
});

const jiraUpdates = await mcp.call('jira', 'search_issues', {
  jql: 'project = ACME AND updated >= -1d',
});

const supportTickets = await mcp.call('zendesk', 'search_tickets', {
  query: 'status:open priority:high',
});

const recentCalls = await mcp.call('gong', 'get_calls', {
  dateRange: { start: yesterday, end: today },
});

// Step 2: Synthesize into brief
const brief = await synthesize({
  slack: slackMessages,
  jira: jiraUpdates,
  support: supportTickets,
  calls: recentCalls,
});

// Step 3: Return artifact (no external writes)
return { type: 'daily_brief', content: brief };
```

### Workflow Pattern 2: Evidence Gathering

**Use case**: PRD drafting, feature research

**Pattern**: Agent searches across sources for a specific topic

```typescript
// Workflow: Feature Evidence Gathering

const topic = 'search date filters';

// Search all sources for relevant mentions
const zendeskEvidence = await mcp.call('zendesk', 'search_tickets', {
  query: topic,
  limit: 50,
});

const gongEvidence = await mcp.call('gong', 'search_calls', {
  keywords: [topic],
  limit: 20,
});

const communityEvidence = await mcp.call('community', 'search_posts', {
  query: topic,
  limit: 30,
});

const analyticsEvidence = await mcp.call('analytics', 'get_search_queries', {
  filter: 'no_results',
  contains: 'date',
});

// Cluster and rank evidence
const clusteredEvidence = clusterByTheme([
  ...zendeskEvidence,
  ...gongEvidence,
  ...communityEvidence,
  ...analyticsEvidence,
]);

return { type: 'evidence_report', themes: clusteredEvidence };
```

### Workflow Pattern 3: Proposal Generation

**Use case**: Creating Jira tickets, Confluence pages, Slack messages

**Pattern**: Agent gathers context, generates proposal, awaits approval

```typescript
// Workflow: Epic Proposal from VoC Theme

// Step 1: Get theme details
const theme = await getVoCTheme('search-frustration');

// Step 2: Gather supporting context
const relatedTickets = await mcp.call('jira', 'search_issues', {
  jql: `labels = "search" AND type = Story`,
});

const competitorInfo = await mcp.call('competitor', 'get_feature', {
  competitor: 'acme',
  feature: 'search',
});

// Step 3: Generate proposal (NOT direct write)
const epicProposal = await mcp.call('jira', 'propose_epic', {
  project: 'ACME',
  summary: 'Search Improvements Q1',
  description: generateDescription(theme, relatedTickets, competitorInfo),
  labels: ['search', 'q1-2026', 'voc-driven'],
  evidence: theme.sources,
});

// Step 4: Return proposal for human review
return {
  type: 'proposal',
  target: 'jira',
  action: 'create_epic',
  payload: epicProposal,
  sources: theme.sources,
};
```

## MCP Tool Design Principles

### Principle 1: Read vs. Write Separation

Clearly separate read operations (autonomous) from write operations (require approval):

```typescript
// Read tools - agent can call freely
'get_issue'      // Fetch a specific issue
'search_issues'  // Search for issues
'get_sprint'     // Get sprint details

// Write tools - always proposals
'propose_issue'  // Propose creating an issue
'propose_update' // Propose updating an issue
'propose_comment'// Propose adding a comment
```

### Principle 2: Scoped Permissions

Tools should request minimal permissions:

```typescript
// Good: Scoped to specific project
await mcp.call('jira', 'search_issues', {
  jql: 'project = ACME', // Limited scope
});

// Bad: Unrestricted access
await mcp.call('jira', 'search_issues', {
  jql: '', // All issues across all projects
});
```

### Principle 3: Rich Context in Responses

Tools should return context, not just data:

```typescript
// Good: Rich context
{
  issue: { key: 'ACME-123', summary: '...' },
  context: {
    project: { name: 'Acme Product', lead: 'Jane' },
    sprint: { name: 'Sprint 23', daysRemaining: 5 },
    reporter: { name: 'John', role: 'Customer Success' },
  },
  related: [
    { key: 'ACME-100', relationship: 'blocks' },
  ],
}

// Bad: Minimal data
{
  key: 'ACME-123',
  summary: '...',
}
```

### Principle 4: Audit-Ready Logging

Every tool call should be loggable:

```typescript
interface ToolCall {
  id: string;
  timestamp: Date;
  tool: string;
  server: string;
  input: Record<string, unknown>;
  output: Record<string, unknown>;
  duration: number;
  context: {
    userId: string;
    tenantId: string;
    jobId: string;
  };
}
```

## Building Custom MCP Connectors

When you need to connect to a tool without an existing connector:

### Step 1: Define the Tool Interface

```typescript
import { z } from 'zod';

const SearchIssuesInput = z.object({
  jql: z.string().describe('JQL query string'),
  maxResults: z.number().optional().default(50),
  fields: z.array(z.string()).optional(),
});

const SearchIssuesOutput = z.object({
  issues: z.array(IssueSchema),
  total: z.number(),
  hasMore: z.boolean(),
});
```

### Step 2: Implement the Server

```typescript
import { BaseMCPServer } from '@pmkit/mcp';

class JiraMCPServer extends BaseMCPServer {
  constructor() {
    super({
      name: 'jira',
      description: 'Jira integration for issue tracking',
      version: '1.0.0',
    });
    this.registerTools();
  }

  private registerTools() {
    // Read tool
    this.registerTool({
      name: 'search_issues',
      description: 'Search Jira issues using JQL',
      inputSchema: SearchIssuesInput,
      outputSchema: SearchIssuesOutput,
      handler: async (input, context) => {
        // Implementation
      },
    });

    // Write tool (proposal)
    this.registerTool(
      createProposalTool(
        'jira_epic',
        'Propose creating a Jira epic',
        EpicInputSchema,
        'jira',
        async (input, context) => ({
          title: `Create Epic: ${input.summary}`,
          preview: formatEpicPreview(input),
          bundle: input,
        })
      )
    );
  }
}
```

### Step 3: Handle Authentication

```typescript
class JiraMCPServer extends BaseMCPServer {
  private async getClient(context: ToolContext) {
    const credentials = await this.getCredentials(context.tenantId, 'jira');
    return new JiraClient({
      host: credentials.host,
      auth: {
        type: 'oauth2',
        accessToken: credentials.accessToken,
      },
    });
  }
}
```

## Security Considerations

### Authentication

- Use OAuth 2.0 for user-context operations
- Store credentials securely (encrypted at rest)
- Implement token refresh handling
- Support tenant isolation

### Authorization

- Implement RBAC for tool access
- Scope permissions to minimum required
- Log all tool calls for audit
- Support permission delegation

### Data Handling

- Don't log sensitive data in tool calls
- Implement data retention policies
- Support data residency requirements
- Handle PII appropriately

## FAQ

**Q: How is MCP different from regular APIs?**
A: MCP provides a standardized interface that AI agents can discover and use. Regular APIs require custom integration code for each tool.

**Q: Can I use MCP with any AI model?**
A: Yes, MCP is model-agnostic. Any AI system that supports tool use can work with MCP servers.

**Q: What about rate limiting?**
A: MCP servers should implement rate limiting and expose it through the protocol. Agents can then respect limits.

**Q: How do I handle errors?**
A: MCP defines standard error formats. Agents should handle errors gracefully and retry when appropriate.

**Q: Is MCP secure for enterprise use?**
A: MCP supports enterprise security patterns (OAuth, RBAC, audit logging). Implementation quality matters.

**Q: What's the performance overhead?**
A: MCP adds minimal overhead. It's essentially a thin wrapper around API calls. The standardization benefits outweigh the small latency cost.
