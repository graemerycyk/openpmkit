# PRD automation that doesn't hallucinate: evidence, assumptions, and open questions

AI can draft PRDs in minutes. The problem is trust. When an AI generates requirements, how do you know they're real? How do you distinguish genuine customer needs from plausible-sounding fabrications?

The answer is evidence grounding: a systematic approach to PRD automation that links every requirement to its source, explicitly labels assumptions, and structures open questions for follow-up.

## The Hallucination Problem in PRD Generation

Large language models are excellent at generating plausible text. That's the problem. A hallucinated PRD requirement looks exactly like a real one:

**Real requirement** (grounded in evidence):
> "Users need date filters in search. 52 customers mentioned this across Zendesk, Gong, and community posts. Enterprise accounts are 3x more likely to mention it."

**Hallucinated requirement** (sounds real, isn't):
> "Users need advanced boolean search operators. Power users frequently request AND/OR/NOT syntax for complex queries."

Both sound reasonable. But the first is backed by data you can verify. The second might be completely made up, or it might be real but from a different product's user base that the model was trained on.

When engineering builds features based on hallucinated requirements, you get:
- Features no one asked for
- Wasted development cycles
- Eroded trust in AI-assisted workflows
- PRDs that require extensive human verification anyway

## The Evidence-Grounded Approach

Evidence-grounded PRD automation follows three principles:

### Principle 1: Every Claim Has a Source

No requirement exists without traceable evidence:

```markdown
## Requirement: Date Range Filters

**Evidence**:
- Zendesk: 28 tickets mentioning date filtering (links)
- Gong: 12 calls where customers mentioned search by date (timestamps)
- Community: 12 posts requesting date filters (links)
- Total: 52 mentions across 3 sources

**Representative Quotes**:
> "I spend more time searching than actually working. If I could just filter by last quarter..."
> - Enterprise customer, Gong call, Dec 15, 2025

> "Please add date range to search. This is table stakes for document management."
> - Community post, Dec 18, 2025
```

### Principle 2: Assumptions Are Explicit

When the PRD makes assumptions, they're labeled:

```markdown
## Assumptions

**A1: Date refers to document creation date**
- Evidence suggests users mean creation date, not modification date
- 8 of 12 Gong mentions specifically said "when it was created"
- **Confidence**: Medium - should validate with user research

**A2: Enterprise customers are primary beneficiaries**
- Enterprise accounts mention date filtering 3x more than SMB
- Likely due to larger document volumes
- **Confidence**: High - clear pattern in data

**A3: Preset ranges are sufficient for MVP**
- No explicit requests for custom date ranges in evidence
- Presets (7 days, 30 days, 90 days) cover common cases
- **Confidence**: Low - may need custom ranges, should validate
```

### Principle 3: Open Questions Are Structured

Unknowns are captured for follow-up:

```markdown
## Open Questions

**Q1: Creation date vs. modification date?**
- Current evidence suggests creation date
- Need to validate with 3-5 user interviews
- Owner: @sarah | Due: Jan 15

**Q2: Performance impact on large result sets?**
- Date filtering may slow queries on accounts with 100K+ documents
- Need engineering assessment
- Owner: @mike | Due: Jan 12

**Q3: Mobile UI for date picker?**
- No specific mobile feedback in evidence
- Need design exploration for touch interfaces
- Owner: @jane | Due: Jan 18
```

## Building an Evidence-Grounded PRD Pipeline

### Step 1: Aggregate Evidence Sources

Connect to all systems where customer signal lives:

| Source | Signal Type | Integration |
|--------|-------------|-------------|
| Zendesk | Support tickets, feature requests | API |
| Gong | Call transcripts, pain points | API |
| Community | Forum posts, feature votes | API |
| Analytics | Usage patterns, no-results queries | API |
| Slack | Internal discussions, customer quotes | API |
| Jira | Existing requests, related work | API |

### Step 2: Extract and Cluster

For a given feature topic, extract relevant evidence:

1. **Search** all sources for related mentions
2. **Filter** for substantive feedback (not just "+1")
3. **Cluster** similar mentions together
4. **Rank** by frequency, recency, and customer segment
5. **Extract** representative quotes with attribution

### Step 3: Generate with Constraints

When generating PRD content, constrain the AI:

```markdown
## Generation Prompt Structure

You are generating a PRD section. Follow these rules:

1. ONLY include requirements that appear in the provided evidence
2. For each requirement, cite specific sources with links/timestamps
3. If you're uncertain about something, label it as an ASSUMPTION
4. If information is missing, add it to OPEN QUESTIONS
5. Do not invent customer quotes or statistics
6. If the evidence is insufficient, say so explicitly

Evidence provided:
[structured evidence from Step 2]

Generate the following PRD section:
[section template]
```

### Step 4: Human Review and Refinement

AI generates the draft; humans verify and enhance:

- **Verify sources**: Click through to confirm quotes are real
- **Validate assumptions**: Mark confidence levels
- **Add context**: Include strategic considerations AI can't know
- **Refine requirements**: Adjust based on product judgment
- **Assign open questions**: Ensure follow-up happens

## Common Pitfalls and How to Avoid Them

### Pitfall 1: Thin Evidence

**Problem**: PRD cites "customer feedback" without specifics.

**Solution**: Require minimum evidence thresholds:
- At least 5 mentions for P0 requirements
- At least 3 sources for major features
- At least 2 direct quotes per requirement

### Pitfall 2: Confirmation Bias

**Problem**: Only including evidence that supports the desired feature.

**Solution**: Actively search for counter-evidence:
- Are there customers who don't want this?
- What are the risks or downsides mentioned?
- What alternatives have customers suggested?

### Pitfall 3: Stale Evidence

**Problem**: PRD cites feedback from 18 months ago.

**Solution**: Weight recent evidence more heavily:
- Last 30 days: Full weight
- 30-90 days: 75% weight
- 90-180 days: 50% weight
- 180+ days: Flag as potentially stale

### Pitfall 4: Missing Segments

**Problem**: Evidence only from one customer segment.

**Solution**: Break down evidence by segment:
- Enterprise vs. SMB
- Industry verticals
- Use case types
- Geographic regions

### Pitfall 5: Unverified Quotes

**Problem**: Quotes that sound real but can't be traced.

**Solution**: Every quote must have:
- Source system (Zendesk, Gong, etc.)
- Unique identifier (ticket #, call ID)
- Date
- Customer attribution (anonymized if needed)

## FAQ

**Q: What if there isn't enough evidence for a feature I know we need?**
A: Document what evidence exists, label the rest as assumptions, and plan validation. Sometimes product intuition is right, but it should be explicit, not hidden.

**Q: How do I handle conflicting evidence?**
A: Document both sides. "Enterprise customers want X, SMB customers want Y." Let the prioritization discussion resolve it.

**Q: What about innovative features customers haven't asked for?**
A: Frame them differently. Instead of "customers requested," use "based on usage patterns" or "strategic initiative." Be honest about the evidence basis.

**Q: How long should evidence gathering take?**
A: With good tooling, 30-60 minutes for a well-scoped feature. The AI does the aggregation; you do the verification.

**Q: What if engineering disputes a requirement?**
A: Show the evidence. "Here are 28 tickets and 12 call mentions." Evidence-grounded PRDs are easier to defend.

**Q: Should I include negative evidence (reasons not to build)?**
A: Yes. A complete PRD acknowledges risks and trade-offs. This builds trust and leads to better decisions.
