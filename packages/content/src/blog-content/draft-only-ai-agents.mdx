# Draft-only AI agents: how to get autonomy without losing control

The promise of AI agents is autonomy; systems that can execute complex workflows without constant human intervention. But in enterprise environments, autonomy creates anxiety. What if the agent writes something wrong to Jira? What if it sends an embarrassing Slack message? What if it makes a commitment to a customer?

The draft-only pattern solves this tension. It gives agents the freedom to work independently while ensuring humans approve every external action.

## The Control Problem with AI Agents

Traditional AI agents operate on a simple loop: observe, decide, act. The agent sees the current state, decides what to do, and executes the action. This works well for games and simulations, but it's dangerous in enterprise environments.

Consider what could go wrong:

- An agent updates a Jira ticket with incorrect information
- An agent sends a Slack message with a hallucinated commitment
- An agent creates a Confluence page with confidential data exposed
- An agent triggers a workflow that notifies customers prematurely

Each of these actions is **irreversible** or **expensive to undo**. Once a Slack message is sent, it's seen. Once a customer is notified, expectations are set.

The instinct is to add more guardrails: more rules, more constraints, more approval gates. But this defeats the purpose of agents. If every action requires approval, you've just built a very complicated copilot.

## The Draft-Only Pattern

Draft-only agents take a different approach. Instead of executing actions directly, they produce **proposals**; drafts that describe what they would do if approved.

```
Agent Workflow:
1. Gather data from connected systems (READ)
2. Synthesize insights and make decisions (THINK)
3. Generate proposals for external actions (PROPOSE)
4. Wait for human approval (WAIT)
5. Execute approved proposals (WRITE)
```

The key insight: **steps 1-3 are completely autonomous**. The agent can read from any connected system, process information, and generate sophisticated proposals without any human intervention.

Only step 5; the actual write; requires approval.

## What Counts as a "Write"?

In the draft-only pattern, we distinguish between:

**Reads** (autonomous):
- Fetching Jira tickets
- Reading Slack messages
- Pulling Gong transcripts
- Querying analytics data

**Internal operations** (autonomous):
- Synthesizing information
- Generating artifacts
- Storing drafts
- Logging actions

**Writes** (require approval):
- Creating or updating Jira tickets
- Sending Slack messages
- Publishing Confluence pages
- Triggering external workflows

The agent can do unlimited reads and internal operations. It only needs approval for writes.

## How Proposals Work

When an agent wants to perform a write, it creates a proposal:

```json
{
  "id": "proposal-12345",
  "type": "jira.create_ticket",
  "target": {
    "project": "ACME",
    "type": "Story"
  },
  "payload": {
    "summary": "Add date filter to search",
    "description": "Based on VoC analysis, users need...",
    "priority": "High",
    "labels": ["search", "q1-2026"]
  },
  "sources": [
    "zendesk-ticket-789",
    "gong-call-456",
    "community-request-123"
  ],
  "rationale": "This addresses the #1 customer pain point...",
  "createdAt": "2025-12-25T10:30:00Z"
}
```

The proposal includes:
- **What**: The exact action to be taken
- **Where**: The target system and location
- **Why**: The rationale and source evidence
- **When**: Timestamp for audit purposes

The PM reviews the proposal, can modify it if needed, and either approves or rejects it.

## Benefits of Draft-Only

### 1. True Autonomy for Data Work

The agent can work completely independently on the hard part: gathering and synthesizing information. A daily brief agent can:

- Pull from 5 different systems
- Process 100+ data points
- Generate a polished summary
- Identify action items and priorities

All without any human intervention. The PM just reviews the final artifact.

### 2. Audit Trail by Default

Every proposal is logged. This creates a complete audit trail of:
- What the agent wanted to do
- Why it wanted to do it
- What evidence it used
- Whether it was approved or rejected
- Who approved it and when

This audit trail is invaluable for compliance, debugging, and improving agent behavior.

### 3. Safe Experimentation

Because writes require approval, you can experiment with agent capabilities safely. Want to try a new PRD template? The agent will generate drafts, but nothing goes to Jira until you approve it.

### 4. Graceful Degradation

If the agent makes a mistake, you catch it before it affects external systems. A hallucinated feature request stays as a proposal; it never becomes a real Jira ticket.

### 5. Human Judgment Where It Matters

The draft-only pattern keeps humans in the loop for decisions that matter: what to communicate, what to commit to, what to prioritize. The agent handles the grunt work; the PM handles the judgment calls.

## Implementation Considerations

### Batching Proposals

For workflows that generate many writes, batch proposals together. A sprint planning agent might generate 10 Jira tickets; present them as a batch for review rather than 10 individual approvals.

### Proposal Expiration

Proposals should expire after a reasonable period. A daily brief proposal from last week isn't useful today. Set TTLs based on the workflow.

### Approval Delegation

In some cases, you might want to delegate approval authority. A senior PM might pre-approve certain types of proposals (e.g., "approve all label updates") while requiring review for others.

### Proposal Modification

Allow humans to modify proposals before approval. The agent's draft is a starting point, not a final answer.

## Common Objections

**"This adds friction to the workflow."**

Yes, but it's the right friction. The agent handles 90% of the work (gathering, synthesizing, drafting). The human handles 10% (reviewing, approving). That's a good trade.

**"What about time-sensitive actions?"**

For truly time-sensitive workflows, you can implement auto-approval rules with constraints. For example: "Auto-approve Slack notifications to internal channels, but require approval for customer-facing messages."

**"Won't this slow down adoption?"**

Actually, the opposite. Draft-only builds trust faster because stakeholders know the agent can't cause damage. Teams are more willing to give agents access to systems when writes require approval.

## Checklist: Is Your Agent Draft-Only?

Use this checklist to evaluate whether an agent follows draft-only principles:

- [ ] Agent can read from connected systems without approval
- [ ] Agent generates proposals for all external writes
- [ ] Proposals include source evidence and rationale
- [ ] Humans can modify proposals before approval
- [ ] All proposals are logged for audit purposes
- [ ] Rejected proposals are tracked with reasons
- [ ] No writes happen without explicit approval

## FAQ

**Q: What about internal writes like logging?**
A: Internal operations (logging, storing drafts, updating internal state) don't require approval. Only external writes to connected systems need approval.

**Q: Can I auto-approve certain types of proposals?**
A: Yes, but carefully. Auto-approval should be limited to low-risk, reversible actions. Always log auto-approved actions for audit purposes.

**Q: How do I handle urgent situations?**
A: For urgent workflows, you can implement expedited approval paths or pre-authorized action types. The key is making this explicit rather than bypassing the pattern entirely.

**Q: What if the agent generates too many proposals?**
A: Batch related proposals together and provide summary views. Good UX for proposal review is essential for the pattern to work at scale.
