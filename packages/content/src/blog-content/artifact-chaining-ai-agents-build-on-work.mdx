# Artifact Chaining: How AI Agents Build on Their Own Work

Most AI tools treat every interaction as a blank slate. You ask a question, get an answer, and start over. But real product work is cumulative. A VoC report informs a PRD. A PRD shapes a prototype. A prototype validates assumptions that refine the roadmap.

What if your AI agent could chain these artifacts together, building compound value over time?

## The Problem with Stateless AI

Traditional AI assistants have no memory. Each prompt exists in isolation:

- **Monday**: You ask ChatGPT to analyze customer feedback. It produces a summary.
- **Tuesday**: You ask it to draft a PRD for a feature. It has no idea about Monday's analysis.
- **Wednesday**: You want a prototype. You're copying and pasting context again.

Every interaction requires you to re-establish context. The AI never learns from its own outputs. You become the integration layer, manually threading insights from one task to the next.

This is exhausting. And it's why "AI productivity gains" often feel smaller than promised.

## Artifact Chaining: A Different Approach

Artifact chaining means AI agents can reference and build upon their previous outputs. When openopenpmkit generates a VoC report, that report becomes a first-class data source for subsequent jobs:

```
VoC Report → PRD Draft → Prototype → Release Notes
     ↓           ↓           ↓
  Evidence    Features    Validation
```

Each artifact in the chain adds context and reduces hallucination. The PRD doesn't invent features—it references specific customer quotes from the VoC report. The prototype doesn't guess at requirements—it implements the PRD's acceptance criteria.

## How It Works in pmkit

pmkit treats its own artifacts as a data source, just like Jira, Slack, or Gong. The openpmkit MCP server exposes tools for querying artifacts:

### Available Tools

| Tool | Purpose |
|------|---------|
| `get_artifact` | Retrieve a specific artifact by ID |
| `list_artifacts` | List artifacts by type (PRD, brief, VoC report) |
| `search_artifacts` | Semantic search across artifact content |
| `get_recent_artifacts` | Get the most recent artifacts of a type |

### Example: PRD to Prototype

When you run the Prototype Generation job, openpmkit automatically:

1. **Queries recent PRDs** using `get_recent_artifacts`
2. **Extracts requirements** from the PRD content
3. **Generates UI code** that implements those requirements
4. **Links the prototype** back to its source PRD

The prototype artifact includes metadata showing its lineage:

```json
{
  "id": "artifact-proto-001",
  "type": "prototype",
  "title": "Search Filters UI Prototype",
  "sourcePrd": "artifact-prd-001",
  "metadata": {
    "featureName": "Advanced Search Filters",
    "requirements": ["date range filter", "content type filter", "multi-select"],
    "framework": "react-tailwind"
  }
}
```

## Real-World Artifact Chains

### Chain 1: Customer Feedback to Shipped Feature

```
Gong Calls + Zendesk Tickets
         ↓
    Feature Intelligence Job
         ↓
    VoC Report (themes, quotes, evidence)
         ↓
    PRD Draft Job (references VoC report)
         ↓
    PRD (requirements, acceptance criteria)
         ↓
    Prototype Generation Job (references PRD)
         ↓
    Prototype (clickable UI)
         ↓
    User Testing → Refinement
         ↓
    Release Notes Job (references PRD + completed Jira tickets)
         ↓
    Release Notes (customer-facing announcement)
```

Each step builds on the previous. Evidence flows through the entire chain.

### Chain 2: Competitor Move to Roadmap Response

```
Competitor Research Job
         ↓
    Competitor Report (feature gaps, pricing changes)
         ↓
    Roadmap Alignment Job (references competitor report)
         ↓
    Alignment Memo (options, trade-offs, recommendation)
         ↓
    PRD Draft Job (references alignment memo)
         ↓
    PRD (response feature spec)
```

### Chain 3: Daily Operations to Sprint Planning

```
Daily Brief Job (every morning)
         ↓
    Brief Artifacts (blockers, updates, risks)
         ↓
    Sprint Review Job (references week's briefs)
         ↓
    Sprint Review Pack (velocity, learnings, recommendations)
         ↓
    Next Sprint Planning (informed by review)
```

## Why Chaining Reduces Hallucination

AI hallucination often happens when models lack context and fill gaps with plausible-sounding fiction. Artifact chaining addresses this directly:

1. **Grounded Evidence**: Each artifact includes citations to source data
2. **Explicit Assumptions**: When the agent makes assumptions, it flags them
3. **Traceable Lineage**: You can trace any claim back through the chain
4. **Consistent Context**: The agent doesn't "forget" what it learned

When a PRD references a VoC report, the PRD includes direct quotes:

```markdown
## Customer Evidence

> "I spend 20 minutes every morning just finding yesterday's documents"
> — Enterprise Customer, Gong Call 2026-01-03

> "The search is basically useless for anything older than a week"
> — Support Ticket #4521, Zendesk
```

The prototype then implements features that address these specific pain points, not imagined ones.

## Proactive Agents: The Next Step

Artifact chaining enables a more ambitious vision: proactive agents that don't wait for instructions.

Today, you trigger jobs manually. But with artifact chaining, agents can:

- **Detect when a PRD is ready for prototyping** and suggest running the job
- **Notice when VoC themes shift** and recommend updating the roadmap
- **Identify stale artifacts** that need refreshing based on new data

The agent becomes a colleague who remembers everything and notices patterns you might miss.

## Getting Started with Artifact Chaining

If you're using pmkit, artifact chaining is built in. Every job automatically has access to previous artifacts through the openpmkit MCP server.

### Best Practices

1. **Run VoC clustering regularly** to build a foundation of customer evidence
2. **Reference artifacts explicitly** when running PRD jobs
3. **Review artifact lineage** to understand where recommendations come from
4. **Keep artifacts updated** as new data arrives

### Example Workflow

```
Week 1: Run Feature Intelligence → Generate VoC Report
Week 2: Run PRD Draft (auto-references VoC) → Generate PRD
Week 3: Run Prototype Generation (auto-references PRD) → Generate Prototype
Week 4: User testing with prototype
Week 5: Run Release Notes (references PRD + Jira) → Ship
```

## The Compound Effect

Individual AI tasks provide linear value. Artifact chaining provides compound value.

Each artifact makes subsequent artifacts better:
- More accurate (grounded in evidence)
- Faster to generate (context already established)
- Easier to review (traceable lineage)
- More consistent (same source of truth)

Over time, your artifact library becomes a knowledge base that accelerates every PM workflow.
